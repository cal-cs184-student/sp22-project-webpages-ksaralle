Part I: Technical Approach

Shader Pipeline

A Shader program is what defines the graphics of the game. The shader folder contains various GLSL files, which are code we wrote for geometry, shadows, and lighting of the visualizations of the game. These made up the main components of the shader pipeline: the gbuffer shaders come first in the pipeline, as they are the main geometry passes, which allows you to render particles, block; The composite shaders then render the texture; Then the outputs are sent to the final shader and get rendered directly to the screen.

Implementation Structure

For this project, we started to write our shader packages from scratch because the code for the shader programs is surprisingly short. We first started by figuring out the basics of pipeline architecture by modifying the shaders file. In our project, we would be mainly focusing on the 2 full-screen passes, composite and final.

First, we implemented the final screen pass. We decided to not do any modifications to the rendering process for the final screen pass, so we simply sampled the color in the fragment shader using the built-in texture coordinates attribute passed from the vertex shader and output that color. Just to check the correctness of our understanding and implementation(because Minecraft would use the default shader if the custom shader does not work), we applied a grayscale to the game by dividing each component of the RGB vector by 3.
